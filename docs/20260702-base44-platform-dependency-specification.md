**Ordered critical-path blockers** (must be resolved before the application can execute independently, in dependency order):

1. **Base44 hosted data platform** (entities store, row-level access control, `base44.entities.*` API) — every page in `src/pages/` reads and writes through this; there is no alternate persistence layer in the codebase. Without a replacement database + API, no page can load or save data.
2. **Base44 authentication/session service** (`base44.auth.*`, `createClientFromRequest`, `appParams.token`) — every protected route depends on `base44.auth.me()` resolving; without it, `ProtectedRoute` and `AuthContext` cannot establish a session, and every backend function that calls `base44.auth.me()` will 401.
3. **`@base44/sdk` client construction** (`src/api/base44Client.js`) — the single point through which entities, integrations, and auth are reached; it requires `appId`, `serverUrl`, and `token`, all currently platform-injected (Section 11).
4. **`@base44/vite-plugin`** (`vite.config.js`) — a build-time dependency; without it the build may fail or lose legacy import resolution, though this is lower severity than 1–3 because it is build-time, not run-time.
5. **Base44-hosted backend functions runtime** (`base44/functions/*/entry.ts`, Deno-based `Deno.serve` handlers) — 15 functions implementing Stripe checkout/portal/webhook, organization/user maintenance, and reporting logic; these must be re-hosted on an alternative serverless/edge runtime before Stripe billing or the admin maintenance utilities function off-platform.
6. **Base44 Core integrations** (`base44.integrations.Core.*`: `InvokeLLM`, `SendEmail`, `SendSMS`, `UploadFile`, `GenerateImage`, `ExtractDataFromUploadedFile`) — consumed throughout `src/pages` and `src/components` (e.g., report generation, referral upload/extraction, assessment creation); each proxies a specific third-party provider that would need direct re-implementation.

**Overall portability verdict**: The application, as committed, is **not portable off Base44** without a substantial rebuild of its persistence, authentication, integration, and function-hosting layers; the UI/component layer (React, Radix, Tailwind, TanStack Query, React Router) is fully portable, but it is inert without the Base44-backed data and auth substrate.

## 3. Classification Legend

**Coupling class** (degree and phase of dependency on Base44):
- `HARD-RUNTIME` — the application cannot execute at run time without the Base44 backend or SDK.
- `HARD-BUILDTIME` — the application cannot be built or bundled without a Base44 build-time component.
- `SOFT` — a Base44 convenience wrapper that could be replaced by a direct third-party implementation without architectural change.
- `PORTABLE` — platform-neutral; no Base44 coupling; requires no action.

**Remediation category** (class of work required to remove the coupling):
- `REBUILD` — must be reconstructed on replacement infrastructure (e.g., a database and API to replace hosted entities).
- `RE-HOST` — existing logic must be redeployed to a non-Base44 runtime (e.g., server-side functions to an alternative edge/serverless host).
- `RE-IMPLEMENT` — must be rewritten against a direct provider (e.g., a platform-provided LLM or email integration).
- `ADAPTER-SHIM` — can be preserved behind a compatibility layer that reproduces the Base44 interface.
- `CONFIG-REWIRE` — requires only reconfiguration, credentials, or environment changes.
- `PORTABLE` — no action required.

## 6. SDK Client Instantiation and Configuration

`src/api/base44Client.js:1-13`:
```js
import { createClient } from '@base44/sdk';
import { appParams } from '@/lib/app-params';

const { appId, serverUrl, token, functionsVersion } = appParams;

export const base44 = createClient({
  appId,
  serverUrl,
  token,
  functionsVersion,
  requiresAuth: false
});
```

- `appId`, `serverUrl`, `token`, `functionsVersion` are all sourced from `src/lib/app-params.js` (D-08), which in turn resolves them, in priority order, from (a) a URL query parameter, (b) a `localStorage` key of the form `base44_<snake_case_param>`, (c) a `import.meta.env.VITE_BASE44_APP_ID` / `import.meta.env.VITE_BASE44_BACKEND_URL` build-time default (for `appId`/`serverUrl` only), or (d) `null`.
- `token` has no build-time default; its sole provenance is the `access_token` URL query parameter (`src/lib/app-params.js:41`, `removeFromUrl: true`), which is appended to the app URL by the Base44 platform's own login redirect flow. There is no path in this repository by which `token` is populated off-platform.
- `requiresAuth: false` is passed to `createClient`, meaning the SDK client itself does not enforce authentication; authentication gating is instead performed application-side in `AuthContext` (D-06) and `ProtectedRoute` (D-07).
- Provenance summary: every configuration value consumed by `createClient` is either platform-injected (URL parameters written by the Base44 builder/runtime) or a Vite build-time environment variable that would need to be independently supplied (`VITE_BASE44_APP_ID`, `VITE_BASE44_BACKEND_URL`) to point at a replacement backend.

## 7. Data Layer and Entity Bindings

Backing store: the Base44-hosted entity store, reached exclusively through `base44.entities.<EntityName>.<operation>()` (e.g., `.list()`, `.filter()`, `.create()`, `.update()`, `.delete()`, `.bulkCreate()`), as documented in the platform SDK. No entity is read or written through any mechanism other than this SDK surface anywhere in the inspected source (`src/pages/*.jsx`, `src/components/**/*.jsx`, `base44/functions/*/entry.ts`).

Row-level security: no explicit row-level-security (RLS) rule declarations exist in any inspected `.jsonc` entity file (schemas contain only `type`, `properties`, `required` — no `access`/`rls` keys were present in any schema body reviewed). Authorization is instead enforced ad hoc in application code, e.g., `base44/functions/getComorbidityReport/entry.ts:8` (`if (user?.role !== 'admin') return 403`) and client-side role checks in `src/Layout.jsx` (`currentUser?.role === "admin"`). This means RLS is **not** a platform-declared feature in this codebase; it is bespoke authorization logic that happens to run against a Base44-hosted store, and it must be reproduced identically (not merely "migrated") in any replacement backend.

Migration action required for every entity: REBUILD — provision an equivalent collection/table in a replacement store reproducing every declared field and constraint, and re-implement the SDK-shaped CRUD interface (`.list`, `.filter`, `.create`, `.update`, `.delete`, `.bulkCreate`, `.bulkUpdate`, `.updateMany`, `.deleteMany`) so that the unmodified call sites in `src/pages/*` continue to function, or (alternative path) rewrite every call site to a new client API.

Data export mechanism: no explicit export function exists in the repository; export would require either (a) the Base44 platform's own data-export tooling (outside this repository's scope, hence UNKNOWN — requires manual verification of what export capability the platform interface offers), or (b) iterating `base44.entities.<Name>.list()` for every entity while still on-platform and serializing the results before decommissioning.

## 10. Authentication and Authorisation

- **Flow**: On load, `AuthProvider` (`src/lib/AuthContext.jsx:16-18,20-88`) calls `checkAppState()`, which first queries the Base44 public-settings endpoint (`GET {serverUrl}/api/apps/public/prod/public-settings/by-id/{appId}`, using a hand-constructed `createAxiosClient` from `@base44/sdk/dist/utils/axios-client`, `AuthContext.jsx:27-34,37`) to determine whether the app requires authentication or the requesting user is unregistered. If a bearer `token` is present (from `appParams.token`, D-08), it then calls `checkUserAuth()` (`AuthContext.jsx:90-111`), which invokes `base44.auth.me()` to fetch the current user and set `isAuthenticated`.
- **Session/token handling**: The bearer token itself is never generated by this repository; it is expected to arrive via the `access_token` URL query parameter (stripped from the URL and persisted to `localStorage` under `base44_access_token`, per `src/lib/app-params.js:9-35,41`). Logout (`AuthContext.jsx:113-124`) delegates entirely to `base44.auth.logout(...)`, which the SDK documentation states handles token cleanup and redirect. Login redirection (`AuthContext.jsx:126-129`) delegates to `base44.auth.redirectToLogin(...)`.
- **Route protection**: `src/components/ProtectedRoute.jsx:11-30` reads `isAuthenticated`, `isLoadingAuth`, `authError` from `useAuth()` and renders `<Outlet />` only when authenticated; it renders `UserNotRegisteredError` for the `user_not_registered` error type and the caller-supplied `unauthenticatedElement` (typically a redirect to `/login`) otherwise. This component is itself platform-neutral (D-07) but is wholly dependent on `AuthContext`'s Base44-backed state.
- **Coupling to the built-in `User` entity**: Application-level role checks (e.g., `src/Layout.jsx`'s `currentUser?.role === "admin"` gating of the "Admin Settings"/"Assessment Audit"/"Analytics" sidebar links, and `base44/functions/getComorbidityReport/entry.ts:8`'s `user?.role !== 'admin'` 403 check) rely on the platform's built-in `User.role` field. There is no independently declared `User.jsonc` schema in this repository (the `User` entity is platform built-in per the system's data model), so its field set (`id`, `created_date`, `full_name`, `email` read-only; `role` and custom fields editable) is authoritative only within Base44.
- **Row-level-security enforcement**: As noted in Section 7, no declarative RLS exists; all authorization is procedural application code coupled to `base44.auth.me()`'s returned `role` field.
- **Replacement provider required**: A self-hosted identity provider (or third-party auth-as-a-service) issuing bearer tokens verifiable by a replacement backend, plus a re-implementation of the `/api/apps/public/prod/public-settings/by-id/{appId}` public-settings contract and the `auth.me()`/`auth.logout()`/`auth.redirectToLogin()`/`auth.updateMe()` method contracts, since `src/lib/AuthContext.jsx` and several functions (`syncStripeSubscription`, all admin functions) are written directly against that method surface.
